### Notable findings from source audit

1. **`rag_engine.py` is the most reusable file in the codebase.** It contains two production-quality system prompts (SYSTEM_PROMPT_CHAT, SYSTEM_PROMPT_OSCE), a well-implemented `retrieve_cases()` function with embedding cache, and `start_osce_session()` with proper case seeding logic. All of this migrates directly into skills.

2. **`osce_scorer.py` is already structured like a skill.** It has a single public function, structured JSON output, temperature=0.1 for determinism, a participation guard, and graceful error handling. It is essentially a completed EvaluationSkill engine.

3. **`database.py` is production-quality and complete.** The OSCE lock, stats queries, session tracking, and topic logging are all implemented correctly. The `get_student_stats()` function returns everything needed to power a StudyPlannerSkill — including `weak_areas`, `top_topics`, and `recent_osce` — without any new queries.

4. **`services/osce_service.py` already implements the OSCE state machine.** The `start_case()` / `turn()` / `finish_case()` pattern is exactly the interface the new `OSCEExaminerSkill` needs. Only the persistence layer needs to change (web_sessions → agent session memory).

5. **The system has NO security layer as a named module.** Safety is handled by system prompt instructions only (STRICT RULES in SYSTEM_PROMPT_CHAT and SYSTEM_PROMPT_OSCE). There is no input sanitizer, no output filter, no PII guard, and no injection detection. This is GAP-C3 and must be built from scratch.

6. **There is no agent controller.** The routing logic in `telegram_bot.py` is a series of if-statements checking `session["mode"]`. There is no intent classification, no skill registry, no structured routing. The web layer has no routing concept at all — it exposes separate endpoints instead.

7. **The i18n system (Babel, English/Russian) is an unexpected asset.** While not required for the competition, it adds real-world polish and demonstrates production thinking. It should be retained in the new codebase.

8. **`config.py` already uses `.env` correctly.** No hardcoded secrets. `API_SECRET_KEY` has a warn-worthy default but is not committed with real values. The secret hygiene gap (GAP-D4) is mostly already closed — only `.env.example` is missing.

## 3. Components to Reuse with Refactoring

These components contain the core logic that must be preserved, but their structure, coupling, or interface must change to fit the agent architecture.

### `rag_engine.py` → split into three components

**What to keep:**
- `SYSTEM_PROMPT_CHAT` → moves to `surgmentor/skills/case_retrieval_skill.py` as the retrieval skill's instruction set
- `SYSTEM_PROMPT_OSCE` → moves to `surgmentor/skills/osce_examiner_skill.py` as the examiner's instruction set
- `retrieve_cases()` function + `_embed_query()` + embedding cache → moves to `surgmentor/rag/retrieval_tool.py` as the `search_vector_store` tool
- `build_context()` → moves to the same retrieval tool module
- `start_osce_session()` → core logic moves to `OSCEExaminerSkill.init()`, case loading stays the same
- `chat()` function's OSCE branch → becomes `OSCEExaminerSkill.turn()`
- `chat()` function's chat branch → becomes `CaseRetrievalSkill.run()` + LLM call pattern

**What to change:**
- Remove the ChromaDB connection at module import time (moves to tool initialization, not module-level side effect)
- Remove the `_ALL_CASES` load at import time (lazy load in skill init)
- Separate retrieval from LLM invocation — they are currently one function call; in the skill system they are two distinct steps

**Estimated refactoring effort:** M (2–3 days, careful extraction needed)

---

### `osce_scorer.py` → `EvaluationSkill` engine

**What to keep:**
- Entire `score_osce_session()` function logic, including participation guard, scoring prompt, JSON parsing, score clamping, and fallback handling
- `MIN_STUDENT_TURNS` constant
- The structured output format: `{score, feedback, strong_areas, weak_areas, teaching_point}`

**What to change:**
- Wrap in a `EvaluationSkill` class with a `run(context_bundle)` interface
- Add a `SessionEvaluation` dataclass as the typed return value (replacing the raw dict)
- Add `eval_log.jsonl` writing after each session is scored
- The scoring prompt can be extended to also output a `study_recommendations` field, enabling a lightweight StudyPlannerSkill without a separate LLM call

**Estimated refactoring effort:** S (1 day)

---

### `database.py` → `surgmentor/memory/db_store.py`

**What to keep:**
- All existing table schemas and query functions (100% correct, production-quality)
- `get_student_stats()` with its weak_areas aggregation — this is the StudyPlannerSkill's data source
- OSCE lock functions
- Session tracking functions
- Topic logging

**What to change:**
- Add a new `agent_sessions` table for platform-agnostic session state (replaces `web_sessions` which is web-specific)
- Move `DB_PATH` to be configurable via `config.py` rather than hardcoded in the file
- Remove `web_login_tokens` table functions (not needed in the new architecture; keep the table definition for backward compatibility with the Telegram bot)
- The file can be renamed and reorganized but the SQL logic stays identical

**Estimated refactoring effort:** S (1 day, mostly rename and one new table)

---

### `services/osce_service.py` → `OSCEExaminerSkill` state machine

**What to keep:**
- The `start_case()` / `turn()` / `finish_case()` state machine design — this is the correct pattern
- The cross-platform lock logic (claim on start, release on finish/reset)
- The "score previous before starting new" flow in `start_case()`
- The participation guard in `finish_case()`

**What to change:**
- Replace `load_osce_state(sid)` / `save_osce_state(sid)` (web-specific SQLite persistence) with the agent session memory interface
- Remove `OsceLockError` (the new architecture uses a single platform identifier; Telegram bot is kept separate)
- Remove direct imports of `rag_engine` (the skill calls the retrieval tool via the agent controller's tool registry instead)
- Rename class methods to match the skill interface: `init()`, `turn()`, `finish()`

**Estimated refactoring effort:** S–M (1–2 days)

---

### `services/chat_service.py` → `CaseRetrievalSkill` LLM call pattern

**What to keep:**
- The `_build_messages()` function structure — the context injection pattern (system + context block + guard turn + history + user message) is correct and should be preserved
- The streaming variant logic (useful for Gradio's streaming output)
- The `_format_sources()` function

**What to change:**
- Remove direct calls to `rag_engine.chat()` — the skill calls `retrieval_tool.search_vector_store()` and then the LLM separately
- Remove the stateless assumption — the skill receives session history from the controller
- The streaming variant becomes a separate method on the skill, not a separate function

**Estimated refactoring effort:** XS–S (half a day)

---

### `database.get_student_stats()` → StudyPlannerSkill data source

**Finding:** `get_student_stats()` already returns `weak_areas` (aggregated from all past OSCE results), `top_topics`, `recent_osce` scores, and `avg_score`. This is exactly what a StudyPlannerSkill needs to generate a personalized remediation plan — no new database queries are required. The StudyPlannerSkill is therefore cheaper to implement than assumed in the gap analysis.

**Implication:** StudyPlannerSkill can be included in MVP (see Section 10).

## 8. Risk List

|---|---|---|---|

| Agent controller routing errors for edge-case inputs | High | Medium | Add `UNKNOWN` intent fallback with graceful message before launch. Test with adversarial inputs. |
| Gradio streaming incompatibility with controller loop | Medium | Medium | Test streaming in Phase 5 early. Fallback: non-streaming output is fully acceptable for judging. |
| Time overrun on security layer (scope classifier LLM call adds latency) | Medium | Medium | Make scope classification optional via config flag. Rule-based check is sufficient for MVP. |

| Kaggle writeup word count exceeded | Low | Medium | Draft early (Phase 6), check count before final submit. |
| Secrets accidentally written into source files before repo is published | Medium | Critical | All API keys come from `.env` only. No key values appear in any source file in this project. Human step: grep for key patterns before publishing. |

## 9. What Should Be Implemented First

**Strict implementation order — do not skip ahead:**

1. **`surgmentor/rag/retrieval_tool.py`** — Every skill depends on this. Nothing works without case retrieval. Test with a direct query before writing any skill code.

2. **`surgmentor/memory/db_store.py`** — The evaluation skill and study planner write to this. Must exist before they do.

3. **`surgmentor/security/layer.py`** — Build it before the controller, not after. It must be a named, importable module — not a post-hoc add-on. Judges will look for it. Building it first also enforces the discipline that all skill outputs are filtered.

4. **`surgmentor/skills/evaluation_skill.py`** — This is the easiest skill (direct refactor of `osce_scorer.py`) and proves the skill interface works before tackling the more complex skills.

5. **`surgmentor/skills/osce_examiner_skill.py`** — The most complex skill, but the most impressive for demo. Build and test in isolation (init → turn → finish) before plugging into the controller.

6. **`surgmentor/skills/case_retrieval_skill.py`** — Powers the free chat tab. Build after OSCE is working.

7. **`surgmentor/skills/study_planner_skill.py`** — Quick to build given `get_student_stats()` is ready. Build last among skills, before the controller.

8. **`surgmentor/agent/controller.py`** — Build after all skills are independently testable. The controller is a router — it should not contain logic that belongs in skills.

9. **`app.py`** — Build after the controller loop is proven on the CLI first (`run.py`).

10. **`README.md` + code comments** — Write after the system works, not before.

## 10. What Should Be Postponed Until After MVP

### StudyPlannerSkill: Promote to MVP

**Revised verdict:** StudyPlannerSkill should be **included in the MVP**, not deferred. Reason: `database.get_student_stats()` already returns everything the skill needs (`weak_areas`, `top_topics`, `recent_osce`). The skill reduces to one LLM call plus a dataclass return. The effort is S (half a day), and it enables the Student Profile tab in Gradio — which makes the demo substantially more compelling for judges. The original gap analysis estimated this as a stretch based on unknown database state; the audit reveals it is nearly free.
